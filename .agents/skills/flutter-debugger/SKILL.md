## 🛡️ The Iron Law of Debugging
**NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST.** Random fixes waste time and create new bugs. You MUST complete Phase 1 before proposing any code changes.

## 🛠️ The 4 Phases of Systematic Debugging (Workflow)

### Phase 1: Root Cause Investigation (Gather Evidence)
*Use MCP Tools to collect raw data. Do not guess.*
- **Baseline Check**: ALWAYS call `get_runtime_errors` at the start to catch hidden exceptions.
- **Visual Inspection**: Use `set_widget_selection_mode` and `get_widget_tree` to map the exact broken component.
- **Trace Data Flow**: If the UI is wrong but there's no error, use `grep_search` to find where the bad value originates (e.g., from a Provider or API call) before touching the Widget.

### Phase 2: Pattern Analysis
*Understand how it should work before fixing it.*
- **Find Working Examples**: Locate similar working UI components or logic in the same codebase.
- **Compare Differences**: What is different between the working component and the broken one?

### Phase 3: Hypothesis Formulation
*Form a single theory based on Evidence (Phase 1) and Pattern (Phase 2).*
- State clearly: "I think X is the root cause because Y data shows Z."
- Do not propose "bundled" fixes (e.g., changing the widget AND the state manager at the same time). 

### Phase 4: Implementation & Verification
*Surgical execution using MCP.*
- **Pre-Reload Audit**: Run `analyze_files` to ensure no syntax errors were introduced.
- **Apply & Reload**: Apply the fix and run `hot_reload`.
- **Re-Verify**: Check `get_runtime_errors` again to ensure the fix didn't spawn a new error elsewhere.

## 🔍 Troubleshooting (Quick Guide)
- **DTD Failed**: URI expired or app not running. Restart and re-copy URI.
- **Empty Tree**: App not in foreground. Bring simulator to front.
- **Reload Error**: Compilation error detected. Fix analyzer issues first.
